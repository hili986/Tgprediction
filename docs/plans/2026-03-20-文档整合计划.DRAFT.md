# 文档整合计划

> **状态**：DRAFT — 等待用户审阅批准
> **目标**：构建一个"结构不变、内容流动"的文档系统
> **原则**：保留历史脉络，消除冗余，服务两类读者（新人速览 / 深度查阅）

---

## 设计理念

**"河流与河床"模型**：
- **河床（不变）**：目录结构、文档模板、命名规范 — 这些一旦确定就不变
- **河水（流动）**：实验数据、进展状态、最新结论 — 随项目推进自动更新

---

### 结构设计原则

| 原则 | 说明 |
|------|------|
| **一个入口** | `docs/INDEX.md` 是所有文档的唯一入口 |
| **三层深度** | INDEX → 主题目录 → 具体文档，最多 3 层 |
| **读写分离** | `overview/` 面向阅读（精简），`research/` 面向查阅（完整） |
| **历史保留** | 原始调研报告保留完整内容，只做分类和去重 |
| **增量更新** | `03-当前进展与成果.md` 和 `实验总览.md` 是唯二需要持续更新的文件 |

## 二、具体执行步骤

### Step 1: 创建目录骨架

- 创建 `docs/overview/`、`docs/research/foundations/`、`docs/research/methods/`、`docs/research/domain/`、`docs/research/decisions/`、`docs/research/other/`、`docs/experiments/`
- 创建 `docs/INDEX.md`（文档入口）
- 创建 `docs/research/_index.md`（调研索引）

### Step 2: 写 overview/ 三份核心文档

**2a. `01-项目背景与目标.md`**
- 从 `项目总览与研究规划.md` 提取：Tg 定义、项目目标、聚合物类型分析
- 从 `Tg预测研究综述与方案设计.md` 提取：Tg 入门部分（1.1-1.3）
- 这份文档一旦写好**基本不变**，只有目标调整时才改

**2b. `02-技术路线.md`**
- 从 `方案选择.md` 提取：方案 A/B 描述
- 从实验计划提取：Phase 1-5 的设计
- 从 `失败原因分析.md` 提取：为什么选 A 不选 C
- 这份文档**很少变**，只有新方案加入时更新

**2c. `03-当前进展与成果.md`**
- 从 `方案A总结.md` 提取核心数据
- 包含：当前最优指标、已完成/进行中/计划中的实验
- 这是**唯一需要频繁更新的 overview 文档**

### Step 3: 整理 research/ 调研库

**3a. 移动文件到子目录**
- `foundations/`：Tg物理化学原理、聚合物表示方法
- `domain/`：核酸调研、VPD调研
- `decisions/`：失败原因、改进方向、方案选择
- `other/`：统计建模大赛相关

**3b. 合并重叠文档**（最复杂的步骤）

合并策略：**保留原文件，创建合并版，原文件移入 `_archive/`**

| 合并目标 | 来源 | 合并方式 |
|---------|------|---------|
| `传统ML与特征工程.md` | 低成本描述符调研 + ML最佳实践 | 描述符部分取前者，模型/评估取后者 |
| `深度学习与GNN.md` | 深度学习方法调研 + 2 份前沿方法调研 | 以深度学习调研为骨架，补充前沿方法中的增量信息 |

合并后在文件头部标注来源：
```markdown
> 本文档合并自以下调研报告：
> - 低成本计算描述符调研.md (2026-03-08)
> - ML最佳实践调研-Tg预测2024-2026.md (2026-03-15)
> 原始文件保留在 docs/research/_archive/
```

**3c. 为保留的文件添加状态标注**

每份保留文档的头部添加：
```markdown
> **时效性**：[有效/部分过时/已过时]
> **被取代情况**：[无/部分被 XX 取代]
> **推荐阅读**：[必读/参考/归档]
```

**3d. 写调研索引 `_index.md`**
- 按分类列出所有调研文件
- 每个文件附 1 句话摘要 + 推荐阅读级别

### Step 4: 创建实验总览

**`docs/experiments/实验总览.md`**
- 统一编号表（E1-E26）
- 每个实验一行：编号、特征、模型、R²、MAE、关键发现、对应 JSON 文件
- 从 `方案A总结.md` + 各 phase summary 提取

### Step 5: 更新根目录文档

- **CLAUDE.md**：更新"当前执行计划"和"实验结果记录"部分
- **KNOWLEDGE_NOTES.md**：在文件开头添加分类索引（按主题分组的锚点列表）

### Step 6: 创建 README.md

- 30 秒版项目介绍
- 链接到 `docs/INDEX.md` 和 `docs/课题组新人入门指南.md`
- 快速上手命令

## 三、需要做和不需要做的

### 做

| 项 | 理由 |
|----|------|
| 创建 INDEX.md 总入口 | 新人最需要的导航 |
| 提炼 overview/ 三文档 | 让"项目现在是什么样"一目了然 |
| 合并 4 份重叠调研为 2 份 | 消除最大的冗余来源 |
| 分类移动调研到子目录 | 按主题组织比按时间堆积好 |
| 创建实验统一编号表 | 跨 phase 查找实验必备 |
| 保留原始文件到 _archive/ | 不丢失任何信息 |

### 不做

| 项 | 理由 |
|----|------|
| 重写所有调研报告 | 工作量巨大且原文质量已经很高 |
| 改动 results/ 目录结构 | 实验脚本和代码引用了这些路径 |
| 改动代码结构 | 代码组织已经清晰，不需要改 |
| 删除任何原始文档 | 移入 _archive/ 而非删除 |

## 四、不变与流动

### 不变的部分（写好后很少改）

| 文档 | 改动频率 |
|------|---------|
| `docs/INDEX.md` | 仅新增文档时更新目录 |
| `docs/overview/01-项目背景与目标.md` | 几乎不改 |
| `docs/overview/02-技术路线.md` | 新方案加入时改 |
| `docs/research/` 所有调研报告 | 不改（已完成的调研） |
| `docs/research/_index.md` | 新增调研时追加条目 |

### 流动的部分（持续更新）

| 文档 | 更新时机 | 更新方式 |
|------|---------|---------|
| `docs/overview/03-当前进展与成果.md` | 每次实验完成 | 追加结果到表格 |
| `docs/experiments/实验总览.md` | 每次实验完成 | 追加行 |
| `KNOWLEDGE_NOTES.md` | 每次教学时刻 | 追加条目 |
| `CLAUDE.md` | 里程碑时 | 更新状态描述 |

### 维持机制（后续配置）

文档的自动维持需要以下支撑，整理完成后再讨论具体实现：

| 机制 | 作用 | 实现思路 |
|------|------|---------|
| **实验完成 hook** | 自动更新实验总览和进展文档 | PostToolUse hook 检测实验脚本完成 |
| **文档更新提醒** | 检测代码变更是否需要更新文档 | Stop hook 中检查 |
| **定期一致性审计** | 检查文档描述与代码/结果是否一致 | 定期 skill 或 cron |
| **模板系统** | 新调研/新实验自动用模板开始 | 调研报告模板 + 实验记录模板 |

## 五、工作量评估

| 步骤 | 预计工作 | 复杂度 |
|------|---------|--------|
| Step 1: 目录骨架 | 创建目录和空文件 | 低 |
| Step 2: overview 三文档 | 从现有文档提取整合 | 中 |
| Step 3a: 移动分类 | 文件移动 + 路径检查 | 低 |
| Step 3b: 合并去重 | 最耗时，需仔细阅读对比 | 高 |
| Step 3c-d: 标注+索引 | 逐文件添加状态头 + 写索引 | 中 |
| Step 4: 实验总览 | 从各 summary 提取数据 | 中 |
| Step 5: 更新配置 | 小幅修改 | 低 |
| Step 6: README | 新写，内容简短 | 低 |

**建议执行顺序**：Step 1 → Step 6 → Step 2 → Step 4 → Step 3 → Step 5

理由：先建骨架和 README（立即可见的改进），再填充 overview 和实验表（核心价值），最后整理调研库（工作量最大但对日常使用影响较小）。
