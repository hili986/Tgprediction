# GPU 加速力场替代方案调研 — 2026-03-25

## 调研背景

### 瓶颈分析
当前 Phase C (链段物理特征) 使用 RDKit ETKDGv3 生成 3-mer 构象 + MMFF94 力场优化，
计算 Boltzmann N_eff 等 8 个物理特征。性能基准测试结果：

| 分子类型 | 重原子数 | 构象生成 (50 confs) | MMFF 优化 | 总耗时 |
|----------|----------|-------------------|-----------|--------|
| 聚苯乙烯 3-mer | 24 | 1.29s | 1.05s | 2.35s |
| 硅氧烷 3-mer | 12 | 0.19s | 0.39s | 0.58s |
| 聚酰亚胺 3-mer | 120 | 82.85s | 30.59s | 113.44s |

**预估全数据集 (7,486 分子) 总耗时**：
- 小分子为主时：~5 小时 (CPU 单进程)
- 含大分子时：~12-24 小时 (CPU 单进程)
- CPU 多核并行 (16 核)：~1-2 小时

**关键发现**：大分子的瓶颈在构象生成 (82.85s) 而非 MMFF 优化 (30.59s)。
GPU 力场替换只能加速优化阶段，构象生成仍在 CPU 上。

### 数据集元素分布
统一数据集 (7,486 条) 中共包含 **24 种元素**：
- 核心有机：H, C, N, O, F, S, Cl
- 显著少数：**Si** (5.8%, 436 分子), **P** (4.7%, 349 分子), **Br** (1.4%, 107 分子)
- 稀有有机：B (0.1%), Se (0.1%), I (0.2%)
- 金属/准金属：Ca, Cd, Ge, K, Li, Na, Pb, Sn, Te, Zn

## 方案 1: TorchANI (ANI-2x 神经网络势)

- **来源**：Roitberg 组 (UF), JCIM 2020; TorchANI 2.0, JCIM 2025
- **仓库**：https://github.com/aiqm/torchani
- **安装**：`pip install torchani`

**元素覆盖：7 种 (H, C, N, O, F, S, Cl)**
- Si (5.8%): 不支持
- P (4.7%): 不支持
- Br (1.4%): 不支持
- **11.9% 的分子无法处理**

**GPU 加速**：完全 GPU 原生 (PyTorch)，CUDA AEV 加速模块
**速度**：RTX 3090 约 1 秒处理 ~16,000 构象单步 (Auto3D 数据)
**精度**：接近 DFT 水平，构象能量排序远优于 MMFF94
**批量处理**：原生支持 batch (species + coordinates tensor)

**代码示例**：
```python
import torch, torchani

device = torch.device('cuda')
model = torchani.models.ANI2x(periodic_table_index=True).to(device)

# species: (batch, atoms) int; coords: (batch, atoms, 3) float
species = torch.tensor([[6, 6, 1, 1, 1, 1, 1, 1]], device=device)
coords = torch.randn(1, 8, 3, device=device, dtype=torch.float32)
energy = model((species, coords)).energies  # Hartree
```

**结论**：元素覆盖严重不足，**排除**作为通用方案。

## 方案 2: AIMNet2 (via Auto3D) ★ 推荐

- **来源**：Isayev 组 (CMU), Chemical Science 2025
- **仓库**：https://github.com/isayevlab/AIMNet2
- **Auto3D**：https://github.com/isayevlab/Auto3D_pkg
- **安装**：
  ```bash
  pip install Auto3D          # 包含 AIMNet2 引擎
  # 或直接 aimnet2calc:
  pip install git+https://github.com/zubatyuk/aimnet2calc.git
  ```

**元素覆盖：14 种 (H, B, C, N, O, F, Si, P, S, Cl, As, Se, Br, I)**
- Si (5.8%): 支持
- P (4.7%): 支持
- Br (1.4%): 支持
- **覆盖数据集 ~98% 的分子**（排除少量金属配合物）

**GPU 加速**：
- 完全 GPU 原生 (PyTorch)
- GPU vs GFN-FF：**5x 更快** (<=80 原子)
- GPU vs CPU：**~100x 更快**
- 10^5 原子系统：单步 GPU <=0.75 秒，O(N) 缩放

**精度** (来自论文，vs DFT 基准)：
- 扭转势能面：MAE = 0.32 kcal/mol, RMSE = 0.47 kcal/mol
- 对比 GFN2-xTB：MAE = 0.73, RMSE = 1.00 kcal/mol
- 对比 MMFF94：MMFF 与 DFT 弱反相关 (-0.1 to -0.45)，排序能力极差
- **AIMNet2 的相对能量排序远优于 MMFF94**

**Auto3D 端到端工作流**：
```python
from Auto3D import Auto3DOptions, smiles2mols

smiles = ["CC(CC(CCc1ccccc1)c1ccccc1)c1ccccc1"]
config = Auto3DOptions(
    k=50,                       # 保留 top-50 构象
    use_gpu=True,
    gpu_idx=0,
    optimizing_engine="AIMNET", # = AIMNet2 since v2.2.1
)
mols = smiles2mols(smiles, config)
for mol in mols:
    energy = float(mol.GetProp('E_tot'))  # Hartree
```

**aimnet2calc 单点能 (灵活，可用于 RDKit 构象)**：
```python
from aimnet2calc import AIMNet2ASE
import ase, numpy as np

calc = AIMNet2ASE('aimnet2')  # auto GPU detection
# 从 RDKit mol 构建 ASE Atoms
numbers = [atom.GetAtomicNum() for atom in mol.GetAtoms()]
positions = conf.GetPositions()
atoms = ase.Atoms(numbers=numbers, positions=positions)
atoms.calc = calc
energy_ev = atoms.get_potential_energy()  # eV
energy_kcal = energy_ev * 23.0609         # eV -> kcal/mol
```

**局限性**：
- 不支持金属元素 (Ca, Cd, K, Li, Na, Pb, Sn, Zn, Ge, Te)
- 仅训练于中性和单电荷体系
- alpha 阶段的 aimnet2calc，API 可能变化

## 方案 3: GFN2-xTB (半经验量化)

- **来源**：Grimme 组 (Bonn), JCTC 2019
- **仓库**：https://github.com/grimme-lab/xtb
- **安装**：`conda install -c conda-forge xtb tblite-python`

**元素覆盖：全周期表 (Z=1-86)**
- 所有 24 种元素全部覆盖，包括金属

**GPU 加速**：有限 (xtb 6.7+ 有 CUDA 支持，但主要针对 MD)
**速度**：比 MMFF94 **慢 5-20x** (SCF 计算开销)
**精度**：显著优于 MMFF94，略逊于 AIMNet2

**代码示例**：
```python
from tblite.interface import Calculator
import numpy as np

numbers = np.array([6, 6, 1, 1, 1, 1, 1, 1])
positions = coords_angstrom * 1.8897259886  # Angstrom -> Bohr
calc = Calculator("GFN2-xTB", numbers, positions)
res = calc.singlepoint()
energy = res.get("energy")  # Hartree
```

**局限性**：
- CPU 为主，比当前方案更慢
- SCF 收敛偶尔失败
- Windows tblite-python 安装不稳定
- **不推荐用于批量加速场景**

## 方案 4: OpenMM + GAFF2 (GPU 经典力场)

- **来源**：Stanford, OpenMM 社区
- **安装**：`conda install -c conda-forge openmm openmmforcefields`

**元素覆盖**：有机分子全覆盖 (需 AM1-BCC 参数化)
**GPU 加速**：能量评估 ~25x faster (CUDA)
**速度瓶颈**：AM1-BCC 电荷参数化每分子 5-30 秒 (CPU)

**结论**：设置复杂度极高 (RDKit -> SDF -> OpenFF -> GAFF -> OpenMM)，
参数化 CPU 瓶颈抵消 GPU 优势，精度与 MMFF94 同级。**不推荐**。

## 方案 5: MACE-OFF (补充)

- **来源**：Csanyi 组 (Cambridge), JACS 2025
- **元素覆盖**：H, C, N, O, F, P, S, Cl, Br, I (10 种)
- **不支持 Si** — 5.8% 分子无法处理
- 精度极高 (0.9-1.5 meV/atom) 但元素覆盖不足。**排除**。

## 推荐方案

### 首选：AIMNet2 分层策略 + CPU 并行 fallback

**理由**：
1. 14 种元素覆盖 ~98% 数据集
2. GPU 完全加速，A800 上预估 20-50x 加速
3. 构象能量精度远优于 MMFF94 (DFT 级)
4. Auto3D 一体化或 aimnet2calc 灵活集成

**分层 fallback 策略**：
```
分子 -> 检查元素集
  |-- 元素全在 AIMNet2 支持范围  ->  AIMNet2 GPU  [~88%]
  |-- 含金属但 MMFF 可处理       ->  MMFF94 CPU   [~10%]
  |-- 其他                       ->  NaN (跳过)    [~2%]
```

**预估性能对比**：

| 指标 | MMFF94 (CPU 单进程) | MMFF94 (16 核并行) | AIMNet2 (A800 GPU) |
|------|--------------------|--------------------|-------------------|
| 小分子 (24 原子) | ~2.4s | ~0.15s | ~0.05s |
| 大分子 (120 原子) | ~113s | ~7s | ~2s |
| 全数据集 (7,486) | 5-24 h | 20-90 min | **5-20 min** |

### 务实备选：纯 CPU 多核并行 MMFF94

如果 AIMNet2 在 A800 上安装遇困难：
- A800 服务器通常有 64-128 CPU 核心
- 32 核并行 MMFF94 → 全数据集 ~15-45 分钟
- **零额外安装**，RDKit 已就绪
- 精度略差但对 N_eff 的 Boltzmann 排序可能足够
